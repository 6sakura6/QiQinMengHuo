## 1. 本次工作流更新目标

本次更新不是单纯调整排期，而是把 Phase 1 v1.1 的叙事强化成果转化为后续研发可以直接执行的生产流程。

### 1.1 更新原则

| 原工作流倾向 | v1.1 更新方向 |
|---|---|
| 以阶段说明为主 | 改为“阶段目标 + 输入 + 任务 + 输出 + 验收”的执行流程 |
| 7 关平均铺开 | 改为“第 1 关完整纵切优先”，先证明核心体验 |
| 叙事和玩法相对分开 | 改为每关都绑定“孟获心态 → 机制变化 → 演出节点” |
| AI 作为加分项描述 | 改为“本地模板优先，混元 API 增强，必须有 fallback” |
| 美术风格待定 | 明确沿用“像素三国”：像素基底 + 水墨/国风 UI 点缀 |
| 测试集中在后期 | 改为每个 Phase 都有阶段门和验收清单 |

### 1.2 研发总目标

> 在有限黑客松周期内，优先完成一个能够完整表达核心差异化的可玩版本：玩家打败孟获后，游戏不以“击杀 Boss”结束，而是进入“擒获—释放—再战”的叙事循环。

### 1.3 三条硬性约束

1. **先纵切，后铺量。** 先做第 1 关完整闭环，再扩展 7 关。
2. **叙事必须可玩化。** 每一段剧情都要对应战斗、演出、关卡机制或玩家反馈。
3. **AI 不做阻塞项。** 混元大模型是增强层，不允许因为 API 延迟、额度、网络或审核问题阻塞主流程。

## 2. 项目概述与已确认决策

| 字段 | v1.1 确认内容 |
|---|---|
| 游戏名称 | 《七擒孟获》 |
| 游戏类型 | 横版动作射击 + 叙事驱动 |
| 核心参考 | 魂斗罗式横版射击节奏 + 三国“攻心为上”的叙事结构 |
| 世界观 | 公元 225 年诸葛亮南征，围绕七擒七纵孟获展开 |
| 玩家视角 | 汉军无名士卒，作为诸葛亮与孟获博弈的亲历者 |
| 核心 NPC | 诸葛亮：战略导师，不神化；孟获：对手、领袖、最终归心者 |
| 美术风格 | 像素三国：像素风为基底，水墨边框、毛笔标题、低饱和国风配色点缀 |
| 目标平台 | Web 浏览器 |
| 推荐技术栈 | Phaser 3 + TypeScript + Vite + CloudBase 静态托管 |
| 核心循环 | 开场授计 → 横版战斗 → Boss 战 → 擒获 → 释放 → 孟获心态变化 → 下一关 |
| 叙事钩子 | “为什么打赢了还要放走？”这一疑问随 7 关逐步揭晓 |
| MVP 范围 | 第 1 关“西洱河初战”完整纵切 |

### 2.1 产品设计支柱

| 支柱 | 说明 | 落地方式 |
|---|---|---|
| 爽快动作 | 玩家要先觉得好玩，叙事才有承载力 | 移动、跳跃、八方向射击、清晰命中反馈 |
| 擒而不杀 | Boss 战结束不是通关，而是叙事反转 | Boss 被击败后切换为“被擒获”状态，不播放死亡动画 |
| 心态递进 | 孟获不是重复失败，而是逐渐理解 | 每关台词、机制和 Boss 策略都对应不同心理阶段 |
| 小兵见证 | 玩家不是诸葛亮本人，而是理解过程的代入者 | 诸葛亮对玩家解释，玩家内心独白承接疑问 |
| 文化厚度 | 不把南中部族写成单纯反派 | 部落、祖灵、技术、自治诉求和内部差异都进入关卡表达 |
| 技术加分 | AI 生成诸葛亮动态计谋台词 | 基于玩家数据生成一句短建议，同时保留本地模板 |

### 2.2 项目非目标

为避免黑客松范围失控，以下内容不进入首个可展示版本：

- 不做开放世界。
- 不做复杂 RPG 装备养成。
- 不做多人联机。
- 不做大量分支结局。
- 不做全语音配音。
- 不在 Phase 2 同时制作 7 关完整内容。
- 不让 AI 直接决定主线剧情，只生成短句式计谋建议。

## 3. 总工作流：7 个 Phase + 7 个阶段门

v1.1 的工作流核心是：每个 Phase 都必须有明确的“阶段门”。只有通过阶段门，才进入下一阶段，避免前期概念不稳、后期返工失控。

| Phase | 阶段名称 | 核心目标 | 阶段门 |
|---|---|---|---|
| Phase 1 | 产品定义 & 叙事设计 | 完成统一方向和可执行文档 | 文档能直接指导第 1 关纵切制作 |
| Phase 2 | 第 1 关纵切原型 | 用最小内容证明核心体验 | 玩家能完整体验“打败 → 擒获 → 释放” |
| Phase 3 | 资产管线建立 | 建立美术、音频、UI、叙事素材规格 | 第 1 关资源可替换灰盒且不破坏手感 |
| Phase 4 | 7 关全量开发 | 扩展七擒七纵完整关卡 | 每关机制都体现孟获心态变化 |
| Phase 5 | 叙事深化 & AI | 数据化对白、碎片、AI 计谋台词 | 无 AI 时也能完整运行，有 AI 时体验增强 |
| Phase 6 | 集成测试 & 部署 | 性能、难度、兼容性、打磨 | 可在线访问，核心流程无 P0/P1 Bug |
| Phase 7 | 提交 & 答辩 | 包装展示作品价值 | 评审 3 分钟内理解差异化，10 分钟内体验核心循环 |

### 3.1 阶段流转图

```text
Phase 1 文档确定
    ↓
Phase 2 第 1 关纵切：证明“擒而不杀”
    ↓
Phase 3 资源替换：建立像素三国资产规范
    ↓
Phase 4 关卡扩展：按孟获七阶段心态扩展机制
    ↓
Phase 5 叙事深化：对白、碎片、AI 计谋台词数据化
    ↓
Phase 6 集成测试：手感、难度、性能、部署
    ↓
Phase 7 参赛提交：视频、Pitch、说明文档
```

### 3.2 推荐时间分配

| 周期 | 主线工作 | 并行工作 |
|---|---|---|
| Week 1 | Phase 1 收敛文档 | 初步视觉概念、角色轮廓 |
| Week 2 | Phase 2 第 1 关灰盒纵切 | Phase 3 第 1 关关键资产 |
| Week 3 | Phase 2 纵切打磨 + Phase 3 资源替换 | Phase 5 对话数据模板 |
| Week 4 | Phase 4 第 2-4 关开发 | 美术扩展、音效补齐 |
| Week 5 | Phase 4 第 5-7 关开发 | Phase 5 AI / 碎片 / 结局演出 |
| Week 6 | Phase 6 测试、调参、部署 | Phase 7 视频素材预录 |
| 最终周 | Phase 7 提交与答辩 | 修复小问题，不新增大功能 |

## 5. Phase 2 — 第 1 关纵切原型

> 核心目标：用第 1 关完整证明“横版射击 + 擒而不杀叙事反转”的核心体验。

Phase 2 不追求内容量，而追求闭环。只要第 1 关可以完整体验，项目就已经有参赛展示的底盘。

### 5.1 本阶段输入

- GDD v1.1：核心玩法、MVP 范围、系统结构。
- level-narrative v1.1：第 1 擒“西洱河初战”完整节点。
- character-sheets v1.1：诸葛亮、孟获、玩家、老兵的台词风格。
- world-bible v1.1：历史背景、文化表达边界。

### 5.2 第 1 关纵切体验目标

```text
开场：诸葛亮说明“此战所求，非杀戮，乃归心”
  ↓
玩家学习移动、跳跃、射击
  ↓
遇到蛮兵与老兵提示
  ↓
孟获骑象 Boss 登场
  ↓
玩家击败孟获
  ↓
孟获不死亡，而是被擒获
  ↓
诸葛亮释放孟获
  ↓
玩家产生疑问：为什么打赢了还要放？
  ↓
结算：第一擒完成，第二擒解锁
```

## 6. Phase 3 — 美术、音频与叙事资产管线

> 核心目标：不是一次性做完所有美术，而是建立可复用、可扩展、可替换灰盒的资产规范。

Phase 3 与 Phase 2 并行推进。程序做灰盒时，美术和叙事团队先做第 1 关必需资产，并制定 7 关扩展规范。

### 6.1 资产优先级

| 优先级 | 资产类型 | 必做内容 |
|---|---|---|
| P0 | 玩家角色 | 站立、跑、跳、射击、受击、死亡 |
| P0 | 孟获第 1 关 Boss | 骑象、冲锋、受击、被击落、被擒获 |
| P0 | 基础蛮兵 | 巡逻、攻击、受击、死亡 |
| P0 | 第 1 关 Tileset | 草地、河岸、平台、背景山脉 |
| P0 | UI | 血条、弹药、对话框、Boss 血条、结算界面 |
| P1 | 诸葛亮头像/立绘 | 对话框头像、释放过场形象 |
| P1 | 老兵张伯头像 | 战斗提示对话框头像 |
| P1 | 音效 | 射击、命中、跳跃、Boss 冲锋、对话弹出 |
| P1 | BGM | 主菜单、第 1 关战斗、第 1 关擒获过场 |

### 6.2 像素三国资产规范

| 项目 | 建议规范 |
|---|---|
| 角色基础尺寸 | 小兵 32×32 或 48×48；Boss 可用 96×96 或更大 |
| Tile 基础尺寸 | 16×16 或 32×32，优先保持统一 |
| UI 风格 | 像素边框 + 水墨纹理 + 朱砂红强调色 |
| 过场风格 | 静态图文分镜优先，不做重动画 |
| 颜色策略 | 低饱和底色，每关用一个主色调表达情绪 |
| 命名规范 | `角色_动作_方向_帧序号.png` 或英文 snake_case |

### 6.4 叙事资产管线

```text
    ↓ 导入

    ↓ 游戏内触发

```

## 7. Phase 4 — 7 关全量开发

> 核心目标：在第 1 关纵切成功后，把核心循环扩展到七擒七纵完整结构。

Phase 4 的关键不是“做 7 个相似关卡”，而是让每一关的机制都体现孟获心态变化。

### 7.1 关卡开发顺序建议

| 批次 | 关卡 | 目标 | 原因 |
|---|---|---|---|
| 批次 A | 第 2、3 关 | 建立环境机制和敌人变化 | 从水流、伏击开始扩展玩法复杂度 |
| 批次 B | 第 4、5 关 | 建立资源压力和双 Boss | 是难度和叙事转折核心 |
| 批次 C | 第 6、7 关 | 建立机关反制和最终综合战 | 最耗时，应在基础系统稳定后制作 |

### 7.2 七关机制与心态绑定表

| 关卡 | 孟获心态 | 关卡机制 | Boss 机制 | 叙事目的 |
|---|---|---|---|---|
| 第 1 擒 | 傲慢 | 基础教学 | 骑象冲锋 | 让玩家产生“为什么放他走”的疑问 |
| 第 2 擒 | 辩解 | 水流、浮台 | 战船移动、撞击平台 | 孟获把失败归因于地利 |
| 第 3 擒 | 困惑 | 视野遮挡、伏击 | 隐身突袭、陷阱 | 孟获认真布局但仍被识破 |
| 第 4 擒 | 倔强 | 毒雾、火把、周期黑暗 | 毒雾弹、毒虫、洞窟塌方 | 外援加入，孟获仍拒绝承认 |
| 第 5 擒 | 挣扎 | 多波次、精英兵种 | 孟获 + 木鹿双 Boss | 南中联军倾巢而出，失败后裂缝出现 |
| 第 6 擒 | 反思 | 火焰陷阱、机关反制 | 巨型木兽、燃料桶引爆 | 技术底牌被破解，孟获开始理解 |
| 第 7 擒 | 折服 | 全机制综合 | 三阶段 Boss，最终放下武器 | Boss 不再战斗，主动归心 |

### 7.3 每关标准制作模板

每关都按以下结构开发，便于程序、叙事、美术协作：

```text
1. 关卡配置
   - 关卡 ID、标题、色调、音乐、背景
2. 开场叙事
   - 诸葛亮授计 + 孟获当前心理状态
3. 关卡主体
   - 3 个推进段落，每段 1 个新机制或敌人组合
4. 中途对话
   - 老兵提示 / 诸葛亮观察 / 敌将挑衅
5. Boss 战
   - 3 个招式 + 1 个叙事机制
6. 擒获演出
   - 孟获反应 + 诸葛亮回应 + 玩家视角疑问
7. 释放或归心
   - 前 6 关释放，第 7 关主动归心
8. 结算
   - 评价、碎片、下一关解锁
```

### 7.4 关卡完成标准

单关不算“完成”，除非同时满足以下 5 项：

- [ ] 可从开场进入战斗并抵达 Boss。
- [ ] Boss 有至少 3 种可读招式。
- [ ] 擒获后有符合该关心态的孟获台词。
- [ ] 诸葛亮释放/回应台词与“攻心”主题一致。
- [ ] 本关新增机制与孟获心态一致，而非单纯换皮。

### 7.5 阶段门验收

- [ ] 7 关都可从选关界面进入并通关。
- [ ] 每关有独立主题、主色调、敌人配置和 Boss 机制。
- [ ] 7 次擒获的台词没有重复感，孟获心理递进清楚。
- [ ] 第 7 关第三阶段能触发“放下武器”而不是普通死亡。
- [ ] 难度曲线不存在突然断崖式上升。
- [ ] 任一关失败重试流程稳定。

## 8. Phase 5 — 叙事系统深化 & AI 计谋台词

> 核心目标：把 v1.1 的叙事文档转化为游戏内可运行、可维护、可替换的数据内容。

Phase 5 与 Phase 4 并行，但不能等所有关卡做完才写对白。每完成一关灰盒，就应该同步填入对应叙事数据。

### 8.1 叙事内容分类

| 类型 | 使用场景 | 数据文件 | 是否可跳过 |
|---|---|---|---|
| 开场过场 | 每关开始，说明局势 | cutscenes.json | 可跳过，首通建议短延迟 |
| 战斗内提示 | 不打断战斗的短句 | dialogs.json | 可跳过或自动消失 |
| Boss 台词 | Boss 登场、半血、击败 | boss_dialogs.json | 不建议完全跳过 |
| 擒获过场 | Boss 战后核心叙事 | cutscenes.json | 首通不可立即跳过 |
| 释放台词 | 诸葛亮的关键表达 | dialogs.json | 首通不可立即跳过 |
| 历史碎片 | 收集解锁文本 | fragments.json | 可在图鉴中阅读 |
| AI 计谋 | 根据玩家数据生成短建议 | strategy_advice.json / API | 可替换为本地模板 |

### 8.2 对话写作规范

| 角色 | 文风 | 禁忌 |
|---|---|---|
| 诸葛亮 | 温和、克制、洞察力强，文言白话融合 | 不要像预言家，不要空喊大道理 |
| 孟获 | 勇武、自尊、重诺，逐关从怒到静 | 不要只会咆哮，不要变成低智反派 |
| 玩家小卒 | 少说话，多用内心独白承接玩家疑问 | 不要抢诸葛亮和孟获的主戏 |
| 老兵张伯 | 朴实、幽默、战场经验丰富 | 不要现代网络梗过多 |
| 蛮族将领 | 各有身份和价值观 | 不要所有人都同一种口吻 |

### 8.3 AI 动态计谋系统落地方案

AI 系统不负责生成主线剧情，只负责生成“诸葛亮对玩家上一关表现的短建议”。

#### 输入数据

```ts
interface PlayerStats {
  levelId: string;
  clearTimeSec: number;
  deathCount: number;
  damageTaken: number;
  enemiesDefeated: number;
  bossHitRate?: number;
  storyFragmentFound: boolean;
  playStyle: 'aggressive' | 'steady' | 'struggling' | 'explorer';
}
```

#### 输出要求

| 项目 | 限制 |
|---|---|
| 字数 | 30-60 字 |
| 角色 | 必须像诸葛亮说的话 |
| 内容 | 只能给战术提醒或心性提醒 |
| 禁止 | 不剧透后续剧情，不改变主线，不使用现代网络语言 |
| 失败处理 | API 失败时使用本地模板 |

#### 本地 fallback 模板

```json
{
  "aggressive": "你锐气可嘉，然南中地势诡谲。下关切记：勇不可无，躁不可有。",
  "steady": "你能稳步推进，已得用兵之要。前路更险，仍须先观其势，再发其箭。",
  "struggling": "胜败乃兵家常事。南中之险，不在一敌一阵，而在不熟其地。慢行，细看。",
  "explorer": "你善察细节，此为良习。战场之中，往往一处火把、一段足迹，便是破局之机。"
}
```

### 8.4 故事碎片系统

| 碎片 | 解锁关卡 | 内容方向 |
|---|---|---|
| fragment_01 | 第 1 关 | 诸葛亮南征背景 |
| fragment_02 | 第 2 关 | 泸水与南中地理 |
| fragment_03 | 第 3 关 | “攻心为上”的策略来源 |
| fragment_04 | 第 4 关 | 南中部族文化与自治诉求 |
| fragment_05 | 第 5 关 | 七擒七纵为何优于歼灭 |
| fragment_06 | 第 6 关 | 木兽、机关与艺术加工说明 |
| fragment_07 | 第 7 关 | 史书记载与“南人不复反” |

### 8.5 阶段门验收

- [ ] 全部对白都在 JSON/CSV 中管理，不硬编码在关卡逻辑里。
- [ ] 每关至少有开场、战斗中、Boss 前、擒获后、结算 5 类文本。
- [ ] AI API 关闭时，游戏仍可完整运行。

## 9. Phase 6 — 集成测试、调优与部署

> 核心目标：让作品稳定、完整、易试玩，而不是继续堆新功能。

### 9.1 测试优先级

| 优先级 | 类型 | 例子 | 处理原则 |
|---|---|---|---|
| P0 | 阻塞流程 | 不能进入关卡、不能通关、结算卡死 | 必须修复 |
| P1 | 严重体验问题 | Boss 卡住、对话遮挡关键战斗、存档丢失 | 必须修复 |
| P2 | 明显瑕疵 | 文本错别字、音量不平衡、小穿模 | 尽量修复 |
| P3 | 锦上添花 | 粒子更多、转场更美、UI 动效更复杂 | 最终周不做 |

### 9.2 调优指标

| 指标 | 目标 | 观察方式 |
|---|---|---|
| 首屏加载 | 尽量短，避免评审等待过久 | 浏览器 Network / 手动计时 |
| 帧率 | 战斗中稳定流畅 | DevTools / 主观体验 |
| 第 1 关时长 | 8-10 分钟 | 玩测记录 |
| 单关死亡次数 | 第 1 关 0-2 次，第 7 关可更高 | 玩测统计 |
| 对话跳过率 | 首通关键对话不应被大量跳过 | 埋点或观察 |
| 核心记忆点 | 玩测者能复述“打赢还放人” | 试玩后访谈 |

### 9.3 兼容性测试

| 环境 | 必测项 |
|---|---|
| Chrome | 主流程、音频、存档、键盘输入 |
| Edge | 主流程、性能、UI 位置 |
| Safari | 音频播放策略、字体显示、LocalStorage |
| 低性能电脑 | 粒子、背景、Boss 战性能 |
| 不同分辨率 | HUD 和对话框不遮挡关键区域 |

### 9.4 部署流程

```bash

npm run build
# 部署到 CloudBase 静态托管
tcb hosting deploy ./dist -e <your-env-id>
```

## 14. 风险清单与规避策略

| 风险 | 表现 | 影响 | 规避策略 |
|---|---|---|---|
| 范围失控 | 同时做 7 关、AI、全动画、复杂养成 | 高 | Phase 2 只做第 1 关纵切，P2 功能全部可砍 |
| 玩法与叙事割裂 | 玩家只记得射击，不记得七擒七纵 | 高 | 每关 Boss 后强制出现“擒获—释放/归心”演出 |
| 孟获脸谱化 | 每关只会喊“不服” | 高 | 每关给新动机：傲慢、辩解、困惑、倔强、挣扎、反思、折服 |
| 诸葛亮神化 | 变成万能预言机器 | 中 | 台词强调观察和判断，而不是预知未来 |
| 文化表达单薄 | 南中部族被写成低级反派 | 高 | 强化自治、祖灵、技术、义气和内部差异 |
| AI 不稳定 | API 延迟、失败、输出不合适 | 中 | 本地模板 fallback；AI 输出短文本；人工审核关键词 |
| 美术拖慢开发 | 等完整精美资源才集成 | 中 | 灰盒优先，资源可逐步替换 |
| 最终周加功能 | 新功能导致老流程崩溃 | 高 | 最终周只修 Bug、录视频、整理材料 |
| 对话太长 | 玩家频繁跳过 | 中 | 战斗提示短于 20 字，过场控制在 1-4 屏 |
| Boss 难度过高 | 玩家看不到叙事核心就流失 | 高 | 第 1 关必须易通关，最终关才高压 |

## 结语

v1.1 工作流的核心不是“把所有东西都做完”，而是确保每一步都服务于同一个参赛表达：

> **《七擒孟获》不是一款把三国元素贴到横版射击上的游戏，而是把“七擒七纵、攻心为上”做成关卡循环、Boss 机制和玩家情绪体验的叙事动作游戏。**

只要第 1 关纵切能让玩家产生“我打赢了，为什么还要放他走？”这个疑问，项目就已经抓住了最重要的核心。后续 6 关的任务，就是用更复杂的机制、更深的角色心理和更完整的演出，把这个疑问一步步回答清楚。
